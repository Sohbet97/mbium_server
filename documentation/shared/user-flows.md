# Пользовательские сценарии — Mbium

Платформа работает в трёх режимах: **покупатель** (мобильное приложение Flutter), **продавец** (веб-панель) и **администратор** (веб-панель). Этот документ описывает полный путь каждого типа пользователя.

---

## 1. Покупатель (Flutter)

Покупатели работают исключительно через мобильное приложение. Веб-панель для них недоступна.

### 1.1 Регистрация
1. Пользователь открывает приложение и вводит номер телефона + пароль.
2. Система создаёт аккаунт со статусом **STATUS_NOT_ACTIVATED (0)**.
3. На телефон отправляется 6-значный OTP-код.
4. После ввода кода аккаунт переходит в статус **STATUS_ACTIVE (1)**.

### 1.2 Вход
1. Ввод номера телефона и пароля.
2. Если включён 2FA (`is_otp_enabled`): система возвращает `session_id`, пользователь вводит OTP-код.
3. Система выдаёт access-token (JWT) и устанавливает httpOnly cookie с refresh-token.
4. В JWT закодированы: `id`, `_role` (объект с `permissions`), поле `user`.

### 1.3 Просмотр каталога и заказ
1. Просмотр категорий → товаров → карточки товара.
2. Добавление в корзину.
3. Оформление заказа: выбор адреса доставки, способа оплаты.
4. Получение уведомления об изменении статуса заказа.

### 1.4 Отзывы и споры
- Покупатель оставляет отзыв после получения заказа.
- При проблеме открывает спор — модератор рассматривает его в admin-панели.

## 2. Заявка на открытие магазина

Покупатель, желающий стать продавцом, проходит следующий путь:

### 2.1 Первичная заявка
1. Покупатель входит через мобильное приложение (или через веб → `/login`).
2. После входа, если нет активного магазина и нет прав администратора, пользователь попадает на страницу `/pending`.
3. На странице `/pending` отображается статус заявки:
   - **Статус 0** — заявка ещё не подана. Кнопка «Открыть магазин» → `/apply`.
   - **Статус 1** — заявка на рассмотрении. Ожидание решения модератора.
   - **Статус 2** — одобрено (технически). Ожидание активации.
   - **Статус 3** — отклонено. Показывается причина отказа, кнопка «Подать повторно» → `/apply`.

### 2.2 Форма заявки (`/apply`)
Пользователь заполняет:

| Поле | Обязательное |
|---|---|
| Тип магазина | ✓ |
| Название (TM) | ✓ |
| Название (RU) | — |
| Описание | — |
| Телефон | — |
| E-mail | — |
| Адрес | — |
| Файл паспорта (URL) | — |
| Файл патента (URL) | — |
| Bank IBAN | — |
| Видео-презентация (URL) | — |

> **Verified PRO:** Магазины с патентом и IBAN автоматически получают статус `seller_tier = 2` при одобрении.

После отправки: статус меняется на **1** (На рассмотрении), пользователь перенаправляется на `/pending`.

### 2.3 Повторная заявка после отклонения
1. Если заявка отклонена (`verification_status = 3`), форма `/apply` предзаполняется прежними данными.
2. Пользователь вносит правки и отправляет повторно.
3. Статус снова становится **1**.

## 3. Администратор — обработка заявок на магазин

### 3.1 Просмотр заявок
1. Администратор → `/admin/shop-applications`.
2. Таблица показывает все заявки со статусом **1** (На рассмотрении).
3. Колонки: название магазина, владелец, дата подачи, тип, ссылки на KYC-документы, кандидат на Verified PRO.

### 3.2 Одобрение
1. Клик «Одобрить».
2. Система:
   - Устанавливает `verification_status = 2`, `is_active = true`.
   - Создаёт запись `ShopMember(role=OWNER)` для владельца.
   - Отправляет Socket.io уведомление владельцу (тип `112 — NOT_SHOP_APPROVED`).
3. При следующем входе владельца `resolveDestination` определяет `shop.is_active = true` → редирект на `/seller`.

### 3.3 Отклонение
1. Клик «Отклонить» → модальное окно с полем «Причина».
2. Система устанавливает `verification_status = 3`, сохраняет `verification_note`.
3. Владелец видит причину на странице `/pending`.

## 4. Продавец (Seller panel)

После одобрения магазина продавец входит на `/seller`.

### 4.1 Вход
1. Продавец вводит логин/пароль на `/login`.
2. `resolveDestination` проверяет:
   - `user.shop.is_active === true` → `/seller`
   - Есть права администратора (`_role.permissions`) → `/admin`
   - Иначе → `/pending`
3. Seller-middleware на каждый запрос к `/seller/*` проверяет наличие активного магазина (`owner_id + is_active`), прикрепляет `req.shop`.

### 4.2 Дашборд (`/seller`)
- Обзор: выручка, заказы, количество товаров.
- Краткая статистика магазина.

### 4.3 Товары (`/seller/products`)
- Список товаров магазина с фильтрацией и поиском.
- Создание товара: название, категория, описание, цена, изображения.
- Варианты товара (размер, цвет и т.д.).
- Редактирование и удаление.

### 4.4 Заказы (`/seller/orders`)
- Список заказов, принадлежащих магазину.
- Изменение статуса заказа.
- Управление отгрузками (shipments).

### 4.5 Скидки (`/seller/discounts`)
- Создание скидок на товары магазина.
- Указание типа (процент / фиксированная сумма), периода действия.
- Редактирование и удаление.

### 4.6 Выплаты (`/seller/payouts`)
- Просмотр текущего баланса.
- История выплат.
- Запрос на вывод средств.

### 4.7 Профиль магазина (`/seller/shop`)
- Редактирование информации о магазине: название, описание, контакты.
- Просмотр плана подписки (тип, лимиты).

## 5. Администратор (Admin panel)

Доступ определяется полем `_role.permissions` в JWT. Роли и их права:

| Роль | Слаг | Основные права |
|---|---|---|
| Superadmin | `superadmin` | Все права (1–84, 309) |
| Администратор | `admin` | Все, кроме удаления ролей (4) и входа под пользователем (309) |
| Модератор | `moderator` | Отзывы, споры, просмотр товаров/заказов/магазинов |
| Контент-менеджер | `content_manager` | Баннеры, категории, товары, коллекции, медиа |
| Менеджер доставок | `delivery_manager` | Курьеры, просмотр заказов |

### 5.1 Пользователи (`/admin/users`)
- Список пользователей с фильтрацией по статусу, роли, телефону.
- Создание пользователя с назначением роли (`role_id`).
- Редактирование: имя, телефон, email, статус, роль.
- Разблокировка заблокированных аккаунтов.
- Мягкое и принудительное удаление.

### 5.2 Роли и права (`/admin/roles`)
- Список системных и пользовательских ролей.
- Системные роли (метка «system», иконка замка) — недоступны для редактирования и удаления.
- Создание пользовательской роли: название, порядок, матрица прав (Read/Create/Edit/Delete по каждому модулю).
- Редактирование прав существующей пользовательской роли.

### 5.3 Магазины (`/admin/shops`)
- Полный список магазинов с фильтрацией.
- Детальная страница магазина: информация, owner, план подписки, KYC.
- Прямое одобрение/отклонение из детальной страницы.

### 5.4 Каталог
- **Категории** (`/admin/catalog/categories`): дерево категорий, создание, редактирование.
- **Товары** (`/admin/catalog/products`): полный список, детальная страница, создание/редактирование.
- **Коллекции** (`/admin/catalog/collections`): управление вручную подобранными наборами товаров.

### 5.5 Заказы (`/admin/orders`)
- Все заказы платформы.
- Изменение статуса, добавление платежей.

### 5.6 Отзывы (`/admin/reviews`)
- Модерация отзывов: одобрение, скрытие.

### 5.7 Скидки (`/admin/discounts`)
- Платформенные скидки (не привязанные к конкретному магазину).

### 5.8 Баннеры (`/admin/banners`)
- Создание и управление рекламными баннерами.
- Типы баннеров (`/admin/banner-types`).
- Изменение порядка отображения (drag-and-drop через reorder API).

### 5.9 Медиатека (`/admin/media`)
- Загрузка изображений и файлов.
- Привязка медиа к товарам.

### 5.10 Курьеры и доставка (`/admin/delivers`)
- Управление службами доставки.

### 5.11 Планы подписки (`/admin/plans`)
- Создание тарифных планов для магазинов.
- Назначение плана магазину через подписки.

### 5.12 География (`/admin/locations`)
- Страны → Регионы → Города → Районы → Сёла.

### 5.13 Настройки (`/admin/settings`)
- Конфигурация платформы: включение OTP, комиссионная ставка.

## 6. Статусы и их значения

### Статус пользователя
| Код | Название | Описание |
|---|---|---|
| 0 | NOT_ACTIVATED | Зарегистрирован, OTP не подтверждён |
| 1 | ACTIVE | Активен |
| 90 | BLOCKED | Заблокирован (временно) |
| 99 | DELETED | Удалён |

### Статус верификации магазина
| Код | Описание |
|---|---|
| 0 | Заявка не подана / черновик |
| 1 | На рассмотрении (submitted) |
| 2 | Одобрен |
| 3 | Отклонён |

### Уровень продавца (seller_tier)
| Код | Описание |
|---|---|
| 0 | Ожидает классификации |
| 1 | Стандартный продавец |
| 2 | Verified PRO (есть патент + IBAN) |

## 7. Авторизация и токены

- **Access-token**: JWT, живёт согласно `ACCESS_TOKEN_EXPIRE_TIME`. Содержит `id`, `_role` (с `permissions`), `user`.
- **Refresh-token**: httpOnly cookie, ротируется при каждом обновлении. Хранится в таблице `user_sessions` (до 3 сессий на пользователя; старейшая удаляется при превышении лимита).
- **RBAC**: каждый защищённый эндпоинт проверяет `req.user._role?.permissions?.includes(PERMISSION_KEY)`.
- **Seller middleware**: дополнительно проверяет `Shop.is_active + owner_id` на каждый запрос к `/seller/*`.
